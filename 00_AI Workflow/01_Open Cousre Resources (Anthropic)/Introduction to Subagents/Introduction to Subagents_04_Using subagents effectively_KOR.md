Subagent를 만들고 잘 설계하는 방법을 알았습니다. 이제 던질 질문은 다음과 같습니다. Subagent는 언제 실제로 도움이 되고, 언제 오히려 방해가 될까요? 그 차이는 한 가지로 귀결됩니다. 바로 중간 작업이 메인 스레드에 중요한지 여부입니다.

## Subagent가 빛나는 때

Subagent는 탐색(exploration)이 실행(execution)과 분리될 수 있을 때 가장 잘 작동합니다. 작업의 각 단계가 이전 단계에서 발견한 내용에 의존한다면, 그 작업은 메인 스레드에 두는 편이 좋습니다. 하지만 결과만 필요하고 그 과정은 중요하지 않다면, subagent에 위임하세요.

Subagent는 다음과 같은 작업에서 탁월합니다:

- 결과가 필요할 뿐, 결과를 찾기까지의 단계별 진행 과정은 필요 없는 경우
- 탐색 작업이 메인 스레드의 컨텍스트를 어지럽힐 가능성이 큰 경우
- 새로운 관점이나 커스텀 system prompt로부터 이득을 볼 수 있는 작업

## 조사 작업

조사(research)는 subagent의 전형적인 활용 사례입니다. 낯선 코드베이스에서 인증이 어떻게 동작하는지 살펴본다고 가정해봅시다. 메인 스레드는 JWT가 *어디서* 검증되는지를 알아야 하지만, 그 과정에서 살펴본 모든 파일을 일일이 볼 필요는 없습니다.

조사용 subagent는 수십 개의 파일을 읽고, 함수 호출을 따라가고, 다양한 코드 경로를 탐색할 수 있습니다. 그 모든 탐색은 subagent의 컨텍스트에 머뭅니다. 메인 스레드는 다음과 같은 깔끔한 요약을 받습니다:

```js
JWT validation happens in middleware/auth.js line 42,
called from the Express router in route/api.js
```

Subagent가 힘든 일을 도맡았습니다. 메인 스레드는 앞으로 나아가는 데 필요한 것을 정확히 받아갑니다.

## 코드 리뷰

Claude는 코드가 다른 사람이 작성한 것처럼 제시될 때 코드를 더 효과적으로 리뷰합니다. 메인 스레드에서 여러 차례에 걸쳐 기능을 구축한 경우, 같은 스레드에 그 코드를 리뷰하라고 요청하면 종종 빈약한 피드백이 나옵니다. Claude가 그 코드를 만드는 데 관여했기 때문에, 새로운 시선으로 보기 어려운 것입니다.

리뷰어 subagent는 별도의 컨텍스트에서 변경사항을 봅니다. `git diff`를 실행하고, 수정된 파일을 읽고, 코드가 어떻게 작성되었는지에 대한 이력 없이 전문적인 리뷰 기준을 적용합니다. 이러한 분리 덕분에 프로젝트별 리뷰 표준을 subagent의 system prompt에 담아둘 수 있고, 팀 전체에 걸쳐 일관된 리뷰 기준을 보장할 수 있습니다.

## 커스텀 System Prompt

Claude Code의 기본 system prompt는 간결하고 코드 중심적인 응답을 강조합니다. 이는 코딩에는 훌륭하지만, 모든 작업에 적합한 것은 아닙니다.

커스텀 system prompt가 subagent를 메인 스레드보다 확실히 더 낫게 만드는 두 가지 사례를 살펴봅시다:

- **카피라이팅 subagent** -- 톤, 대상 독자, 스타일에 대한 지시사항을 부여하세요. Claude Code의 기본 프롬프트는 간결한 기술 문서 스타일로 기울어지기 쉬운데, 이는 랜딩 페이지나 이메일 캠페인에 어울리는 글쓰기와는 거리가 멉니다. 카피라이팅 subagent에는 목소리와 구조에 대해 완전히 다른 지시사항을 줄 수 있습니다.
- **스타일링 subagent** -- 디자인 시스템 파일을 참조하도록 설정하세요. Subagent가 실행되면 그 파일들이 자동으로 컨텍스트에 로드되어, CSS를 한 줄도 작성하기 전에 이미 색상 변수, 간격 규칙, 컴포넌트 패턴을 파악하고 있게 됩니다.

## Subagent가 해가 되는 때

Subagent를 띄우는 데에는 오버헤드가 따릅니다. Subagent의 작업 과정에 대한 가시성을 잃고, 발견 사항이 요약으로 압축되기 때문입니다. 따라서 메인 스레드가 할 수 없는 일을 subagent가 해줄 때에만 그 오버헤드를 감수할 가치가 있습니다. 주의해야 할 흔한 anti-pattern(안티패턴)이 세 가지 있습니다.

### 전문가 페르소나(Expert Claims)

전문성을 내세우는 subagent는 거의 도움이 되지 않습니다. "당신은 Python 전문가입니다" 또는 "당신은 Kubernetes 전문가입니다" 같은 프롬프트는 Claude가 이미 그 지식을 가지고 있기 때문에 별다른 가치를 더하지 않습니다. 소위 전문가 subagent가 할 수 있는 일 중 메인 스레드가 직접 할 수 없는 일은 없습니다.

### 순차적 파이프라인(Sequential Pipelines)

순차적 subagent 파이프라인은 문제를 일으킵니다. 버그를 재현하는 에이전트, 디버깅하는 에이전트, 수정하는 에이전트로 이어지는 3단계 흐름을 생각해봅시다. 파이프라인은 각 작업이 진정으로 독립적일 때만 잘 동작합니다. 각 단계가 이전 단계에서 발견한 내용에 의존할 때는 실패하는데, 버그 수정은 거의 항상 그런 경우에 해당합니다. 에이전트 간의 인계 과정에서 정보가 유실됩니다.

### 테스트 러너(Test Runners)

테스트 러너 subagent는 필요한 정보를 가리는 경향이 있습니다. 테스트가 실패하면 문제를 진단하기 위해 전체 출력이 필요합니다. 그런데 "tests failed"만 반환하는 subagent를 두면, 직접 실행했다면 그대로 보였을 세부 정보를 얻기 위해 추가 디버그 스크립트를 만들어야 합니다. 실험 결과, 테스트 러너 패턴은 모든 구성 중 성능이 가장 낮은 것으로 나타났습니다.

## 결정 규칙

Subagent를 사용할지 결정할 때는 스스로에게 한 가지 질문을 던져보세요. **중간 작업이 중요한가?**

대답이 "아니오"라면 -- 최종 결과만 필요하다면 -- subagent에 위임하세요. 대답이 "예"라면 -- 진행 과정을 직접 보고 반응해야 한다면 -- 메인 스레드에서 처리하세요.

Subagent를 사용하기 좋은 경우:

- 조사와 탐색
- 코드 리뷰
- 커스텀 system prompt가 필요한 작업

Subagent 사용을 피해야 하는 경우:

- 실제 역량을 더해주지 않는 "전문가" 페르소나
- 각 단계가 이전 단계에 의존하는 다단계 파이프라인
- 디버깅을 위해 전체 출력이 필요한 테스트 실행
