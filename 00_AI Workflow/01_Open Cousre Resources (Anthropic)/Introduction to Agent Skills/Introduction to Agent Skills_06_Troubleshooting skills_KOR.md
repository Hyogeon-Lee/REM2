**학습 내용**

*예상 소요 시간: 15분*

- skill(스킬) validator(검증자)를 사용하여 디버깅 전 구조적 문제 포착하기

## Skill(스킬) 문제 해결하기

*(4분)*

![](https://www.youtube.com/watch?v=YBa1cwaG7is)

Skill(스킬)이 예상대로 작동하지 않으면 문제는 일반적으로 몇 가지 예측 가능한 범주에 속합니다. 이 비디오는 각 범주 - skill(스킬)이 트리거되지 않는 것부터 우선순위 충돌, 런타임 실패까지 - 를 살펴보고 체계적인 문제 해결 접근법을 제공합니다. 또한 skill(스킬) validator(검증자) 도구와 로딩 문제를 진단하기 위해 `claude --debug`를 사용하는 방법을 배웁니다.

#### 핵심 내용

- **skill(스킬) validator(검증자) 도구**로 시작합니다 - 다른 것들을 디버깅하는 데 시간을 소비하기 전에 구조적 문제를 포착합니다
- Skill(스킬)이 **트리거되지 않으면** 원인은 거의 항상 설명입니다 - 실제로 요청을 어떻게 표현하는지 일치하는 트리거 구문을 추가합니다
- Skill(스킬)이 **로드되지 않으면** `SKILL.md`가 skill(스킬) 루트가 아닌 명명된 디렉토리 내에 있는지, 파일 이름이 정확하게 `SKILL.md`인지 확인합니다
- **잘못된 skill(스킬)이 사용되면** 설명이 너무 유사합니다 - 더 구별되게 만듭니다
- **런타임 오류**의 경우 종속성, 파일 권한(`chmod +x`), 경로 구분자(모든 곳에서 슬래시 사용)를 확인합니다

Skill(스킬)이 작동하지 않으면 일반적으로 몇 가지 범주 중 하나에 속하는 문제입니다: skill(스킬)이 트리거되지 않음, 로드되지 않음, 충돌이 있거나, 런타임에 실패함. 좋은 소식은 대부분의 수정이 상당히 간단하다는 것입니다.

## Skill(스킬) Validator(검증자) 사용하기

먼저 시도할 것은 agent skill(스킬) verifier(검증자) 명령입니다. 설치 단계는 운영 체제에 따라 다르지만 `uv`를 사용하는 것이 가장 빠르게 설정하는 방법입니다.

설치 후 skill(스킬) 디렉토리로 이동하거나 어디서나 명령을 실행합니다. Validator(검증자)는 다른 것들을 디버깅하는 데 시간을 소비하기 전에 구조적 문제를 포착합니다.

## Skill(스킬)이 트리거되지 않음

Skill(스킬)이 존재하고 검증을 통과했지만 예상할 때 Claude가 이를 사용하지 않습니다. 원인은 거의 항상 설명입니다.

Claude는 semantic matching(시맨틱 매칭)을 사용하므로 요청이 설명의 의미와 겹쳐야 합니다. 충분한 겹침이 없으면 일치하지 않습니다. 다음을 하세요:

- 실제로 요청을 어떻게 표현하는지에 대해 설명을 확인합니다
- 사용자가 실제로 말할 트리거 구문을 추가합니다
- "이 프로세스를 프로파일하는 데 도와줘", "이게 왜 느린가?", "이를 더 빠르게 만들어줘" 같은 변형으로 테스트합니다
- 어떤 변형도 트리거되지 않으면 그 키워드를 설명에 추가합니다

## Skill(스킬)이 로드되지 않음

Claude에게 "어떤 skill(스킬)이 사용 가능한가?"라고 물었을 때 skill(스킬)이 나타나지 않으면 이러한 구조적 요구사항을 확인합니다:

- `SKILL.md` 파일은 skill(스킬) 루트가 아닌 명명된 디렉토리 내에 있어야 합니다
- 파일 이름은 정확하게 `SKILL.md`여야 합니다 - "SKILL"은 모두 대문자, "md"는 소문자입니다

`claude --debug`를 실행하여 로딩 오류를 봅니다. Skill(스킬) 이름을 언급하는 메시지를 찾습니다. 때로 이것만으로도 문제를 바로 지적합니다.

## 잘못된 Skill(스킬)이 사용됨

Claude가 잘못된 skill(스킬)을 사용하거나 skill(스킬) 간에 혼동되는 것처럼 보이면 설명이 너무 유사합니다. 구별되게 만듭니다. 최대한 구체적이 되면 Claude가 skill(스킬)을 언제 사용할지 결정하는 데만 도움이 되지 않고 - 다른 비슷하게 들리는 skill(스킬)과의 충돌도 방지합니다.

## Skill(스킬) 우선순위 충돌

개인용 skill(스킬)이 무시되면 enterprise(엔터프라이즈) 또는 더 높은 우선순위의 skill(스킬)이 같은 이름을 가질 수 있습니다.

![](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2Fa46l9irobhg0f5webscixp0bs%2Fpublic%2F1771527274%2FSkills6_12.1771527274623.png)

예를 들어, enterprise(엔터프라이즈) "code-review" skill(스킬)이 있고 개인용 "code-review" skill(스킬)도 있다면 enterprise(엔터프라이즈) 것이 매번 이깁니다. 옵션:

1. Skill(스킬)의 이름을 더 구별되는 것으로 변경합니다(일반적으로 더 쉬운 방법)
2. 관리자와 enterprise(엔터프라이즈) skill(스킬)에 대해 대화합니다

## 플러그인 Skill(스킬)이 나타나지 않음

플러그인을 설치했지만 skill(스킬)을 볼 수 없나요? 캐시를 지우고, Claude Code를 다시 시작하고, 다시 설치합니다.

그 이후에도 skill(스킬)이 나타나지 않으면 플러그인 구조가 잘못되었을 수 있습니다. 이것이 validator(검증자) 도구가 정말 가치 있을 때입니다.

## 런타임 오류

Skill(스킬)이 로드되지만 실행 중에 실패합니다. 몇 가지 일반적인 원인:

- **누락된 종속성:** Skill(스킬)이 외부 패키지를 사용하면 설치되어야 합니다. Claude가 필요한 것을 알도록 skill(스킬) 설명에 종속성 정보를 추가합니다.
- **권한 문제:** 스크립트에 실행 권한이 필요합니다. Skill(스킬)이 참조하는 스크립트에 `chmod +x`를 실행합니다.
- **경로 구분자:** 모든 곳에서 슬래시를 사용합니다. Windows에서도 그렇습니다.

## 빠른 문제 해결 체크리스트

- **트리거되지 않음?** 설명을 개선하고 트리거 구문을 추가합니다.
- **로드되지 않음?** 경로, 파일 이름, YAML 문법을 확인합니다.
- **잘못된 skill(스킬) 사용됨?** 서로 다른 설명을 더 구별되게 만듭니다.
- **가려짐?** 우선순위 계층을 확인하고 필요하면 이름을 변경합니다.
- **플러그인 skill(스킬) 누락?** 캐시를 지우고 다시 설치합니다.
- **런타임 실패?** 종속성, 권한, 경로를 확인합니다.
